# Literature review and comparison with SOTA

**Project:** Schabaschkascuhen — a personal job-search pipeline for the German market
(JobSpy fork + native Arbeitsagentur client → Normalizer (qwen3.5:4b) → hard filters → LLM Judge (qwen3:8b) → daily slate → Vlad's labels: ~100 bootstrap + ~10/day).
**Date:** 2026-06-12. **Status:** mandatory deliverable — lit-review + comparison with SOTA.

---

## 1. Review questions

The project plan rests on the chain "scrape → normalization → filters → LLM judge → slate → feedback", while the entire training signal is ~100 one-shot labels plus ~10 labels per day from a single person. From this follow concrete questions the review had to answer **before** the roadmap was frozen:

- **Q1. Reliability of the LLM judge on local small models.** Can qwen3:8b produce calibrated 1–5 scores? Pointwise vs pairwise vs listwise; known biases (position, verbosity) and how to suppress them mechanically; how to measure calibration.
- **Q2. How to spend a budget of ~100 labels.** Which example-selection strategy works under a low budget (active learning "on the cheap"); which label format is more reliable (1–5 scale vs best-worst); does free-form preference elicitation add anything.
- **Q3. A proven job-recommendation architecture.** Which pipeline form is demonstrably effective (RecSys Challenge 2017 = the canonical job-rec benchmark); which embedding backbone to use for a mix of German and English listings; is an ESCO/ISCO taxonomy needed.
- **Q4. What to train at n=100.** Should anything LLM-sized be fine-tuned; frozen embeddings + a classical model vs SetFit vs a neural ranker.
- **Q5. AI-slop.** Is local detection of machine-generated vacancy text feasible, and can it be used to filter hard.
- **Q6. Import-over-build.** Which repositories to fork / import as a library / copy as a pattern / reject — in line with the "write as little as possible ourselves" rule.
- **Q7. Annotation tooling.** Does an off-the-shelf annotation platform (Argilla, Label Studio, doccano, Prodigy) pay off at a volume of 100 one-shot + 10/day, or is xlsx + a micro-FastAPI enough.
- **Q8. Comparison with SOTA patterns.** Where our plan diverges from the patterns in the local knowledge base (claude-code, ARE/Gaia2, aislop) on retry, error taxonomy, judge pinning, slop scoring, and prompt discipline.

### Decisions supported by the literature

1. **The judge protocol is comparative, not pointwise.** At the qwen 4b/8b scale listwise fails and pointwise 1–5 is uncalibrated (PRP, Setwise): setwise tournaments in groups of 3–5 with pick best/worst; if a 1–5 rubric stays for display — calibrate it with anchor examples from Vlad's labels and check kappa against the pairwise outcomes.
2. **Mechanical debias of every judge call:** order randomization and swap with aggregation (position bias up to ~75% preference for the first slot), description length normalization/truncation (verbosity bias), a permanent metric — Cohen's kappa judge-vs-Vlad on held-out labels.
3. **A two-stage judge:** deterministic/binary gate checks of Normalizer fields (location, remote, language, contract type, seniority) **before** any LLM preference comparison — binary criteria don't fit the pairwise paradigm and waste the comparison budget.
4. **A 100-label budget, low-budget-AL style:** cluster bge-m3 embeddings and build slates from typical/representative examples stratified by cluster and board (TypiClust: uncertainty sampling is below random at < a few hundred labels); leave uncertainty/margin for later, but route the judge's low-confidence items into the slates (Personalized Judge).
5. **Labels as best-worst 4-tuples** (pick the best + worst), not 1–5 stars — the same budget, significantly higher reliability, each tuple = 5 pairwise constraints (Kiritchenko & Mohammad); complement with a one-time free-text GATE interview that becomes standing judge context (GATE, Sanner).
6. **A learned ranker at n=100:** frozen bge-m3 features (cosine profile→vacancy, mean/max similarity to liked ones, ESCO match, metadata) → LightGBM or regularized logreg; no LLM fine-tuning, no neural ranker (Grinsztajn, SetFit, MSR RecSys-2017 lessons); the only supported escalation is SetFit-style contrastive tuning of the encoder; choose negatives deliberately (shown-but-skipped is a weak negative).
7. **ESCO/ISCO as a scaffold:** the Normalizer attaches an ESCO occupation + top skills via embedding similarity; use it as slate stratification keys, judge context, and cross-board dedup help — but the matching backbone stays raw bge-m3 proximity (ESCOXLM-R, MELO).
8. **AI-slop as a soft feature only:** a Binoculars-style perplexity-contrast score from two local models at a fixed low-FPR threshold, only for descriptions above a length threshold; never a hard filter (RAID: catastrophic failures under domain shift; reliable short-text detection is closed-source per Jabarian & Imas); combine with cross-posting boilerplate/repetition heuristics.

### Where the evidence base is thin (honest caveats)

- **lmd_classi** — a single author's practitioner benchmark on one dataset (French short texts), not peer-reviewed; we use it as corroborating, not primary, evidence.
- **Personalized Judge (~70% agreement ceiling)** was measured on generic personas; our regime (a rich first-person profile + few-shot from the same person's labels) is closer to the paper's best case, but there are no direct measurements for the job domain — so judge-vs-Vlad kappa stays a mandatory metric, not an assumption.
- **Binoculars on German vacancies** — an extrapolation: both Binoculars and RAID were measured in English, and job-ad boilerplate is out-of-domain; the threshold will have to be revalidated by hand on our sample.
- **Jabarian & Imas** — a working paper (NBER 2025), not yet peer-reviewed.
- **Kiritchenko & Mohammad** — best-worst was measured on sentiment annotation, not vacancy preferences; the transfer is plausible (the same class of subjective scalar-judgment task), but there's no direct check.
- **RecSys Challenge 2017** — scale 900k users vs our n=1; we take only the pipeline form and the cold-start/negatives lessons, not the numbers.
- **Operational note:** two WebSearch queries were rejected by the model's safety layer ("API Error: Fable 5 has safety measures that flag messages on most cybersecurity or biology topics... Request ID: req_011Cbz1RtYTrmEa2HgX4Fwtg" and "...req_011Cbz1YxV3Hxn5ctTC1PRBe"); both were reworded and run, no coverage was lost.

### Gaps in our plan vs SOTA

1. **No grader-identity pinning beyond rubric_version:** JudgeScore stores the model name, but not the ollama digest and not the few-shot set hash — the weekly CV agreement deltas are currently unattributable to (rubric | few-shot | model), violating are/pinned-judge-model.
2. **The CV gate is a single pooled number:** no fold-level STD/SEM, no repeated runs (N=3) of the non-deterministic local judge, no Pass^k analog — violates are/multi-run-variance; a flaky judge can pass the 75% gate.
3. **No tri-state outcomes:** "the judge returned invalid JSON → the card stays normalized" mixes unjudgeable with not-yet-judged in the funnel denominators; there's no harness-fault class separating "the Normalizer broke the card" from "the judge erred" in the weekly agreement metric (are/failure-taxonomy).
4. **slop_flag is binary in the card_json schema** — no severity-weighted findings, no length/density normalization, no smoothing, no tiers (aislop/slop-scoring); long German postings will be biased toward slop on raw signal counts.
5. **Retry is ad-hoc prose** ("retry x2 with backoff" in Extension 7a): no central wrapper, Retry-After is ignored, no jitter, no per-status-code retryability table, no purpose-aware (batch-vs-interactive) budget (claude-code/retry-backoff).
6. **Error logging is unstructured** ("classified errors in the log"): no fixed coarse-enum column + lossless raw-details column in SQLite, so the FSM can't make typed retry decisions and the funnel can't count error classes (claude-code/error-envelope).
7. **The judge few-shot prompt lacks assembly discipline:** no `<example>`+`<commentary>` structure, no negative examples (score-1 / insufficient-evidence), no programmatic interpolation of the tag vocabulary from schema constants, no flag-based swapping of example sets as labels accumulate (claude-code/tool-prompts-fewshot).
8. **profile/rubric.md lacks staleness/supersession discipline:** weekly updates risk append-only growth and contradictory preference facts; there's no mtime-based "N days ago" freshness on the few-shot labels (claude-code/memdir-file-memory).
9. **Canary verdicts are binary min-row assertions** without a closed status enum (dead_scraper / degraded / empty_market / stale_parser) — the measured Google silent-zero signature deserves its own counted failure type, not an alert string (are/failure-taxonomy).
10. **No deterministic short-circuit before LLM normalization** (content-hash reuse for reposts, a min-length floor) — the nightly K-budget pays for repeated work that the measured TBC-9 churn predicts (the cheap-exit discipline of are/hard-before-soft-judge).

### Implementation priority (top-3 by the SOTA-mapping verdict)

1. **The hard-before-soft filter ladder** (are/hard-before-soft-judge, tool_judge.py#L591-L631) + content-hash short-circuits before normalization — most verdicts become deterministic/reproducible, and both LLM budgets are capped to the genuinely-ambiguous remainder.
2. **Pinning the full grader-tuple** (are/pinned-judge-model, configs.py#L27-L59) + fold-level SEM and N=3 verdict-stability right away (are/multi-run-variance) — otherwise the CV gate and all agreement trends are unattributable noise.
3. **A typed tri-state failure taxonomy across the whole FSM** (are/failure-taxonomy, judgment.py#L13-L104 + claude-code/error-envelope) — the funnel becomes counted metrics and the judge's agreement stays honest. Honorable mention: density-smoothed slop scoring with tiers instead of a binary slop_flag (aislop/slop-scoring) — cheap when designing the card schema, expensive to retrofit after the bootstrap labels are collected.

## 6. Summary: what the review changes in the plan

- **Judge: drop listwise slates and raw pointwise 1–5 on qwen3:8b** — move to setwise/pairwise comparisons (groups of 3–5, pick best/worst) with candidate-order randomization and swap, description-length normalization, and 1–5-rubric calibration via anchor examples; the worthwhile calibration metric is Cohen's kappa judge-vs-Vlad on held-out labels (PRP, Setwise, Gu et al.).
- **Split the judge into two stages:** deterministic binary gate checks of hard constraints (location, remote, language, Zeitarbeit, contract type) BEFORE any LLM comparison; field routing via a declarative CheckerType enum (hard vs soft), not if/else (Eugene Yan + are/hard-before-soft-judge).
- **Build hard detectors with FP guards:** the regex German-requirement detector = a broad pattern + an exoneration cascade ("nice to have", "is a plus", "von Vorteil" → not-required), a scope gate of boilerplate sections before matching, a regression gate ≥25/27 on the 27-line spike test set; dedup = an exoneration-first guard stack, fuzzy candidates logged and never auto-merged (aislop/fp-guards).
- **Label collection: best-worst 4-tuples instead of 1–5 stars** (each tuple = 5 pairwise constraints); the first slates TypiClust-style: cluster bge-m3 embeddings, typical examples, stratification by board/role/language; defer uncertainty sampling until a few hundred labels; route the judge's low-confidence verdicts into the next annotation slate (Kiritchenko & Mohammad, TypiClust, Personalized Judge).
- **Add a one-time GATE interview** (the LLM asks Vlad open questions and edge cases) and store the transcript + the written preference profile as standing judge context; the system must rank acceptably zero-shot before any labels (GATE, Sanner).
- **Learned ranker at n=100: frozen bge-m3 + LightGBM/regularized logreg** (cosine profile→vacancy, mean/max similarity to liked ones, ESCO match, metadata); no LLM/cross-encoder fine-tuning; the only supported escalation is SetFit-style contrastive tuning of the encoder; choose negatives deliberately — a shown-but-skipped vacancy is a weak negative (Grinsztajn, SetFit, MSR RecSys-2017).

## 7. v2: aspect-based CV↔JD matching + ML gate + agentic search (implemented 2026-06-14)

### 7.1 Aspect-based matching (bge-m3, `aspects.py` + `features.py`)

**The problem with a single blob cosine:** whole-document cosine resume↔vacancy is a measure of *similarity*, not *fitness*: an "About us" with a similar domain inflates the score regardless of the actual requirements. The SOTA approach for job-skill matching is **asymmetric must-have coverage** (do the candidate's skills cover the JD's specific requirements?), not a symmetric blob (Xu et al. 2023 "Towards Comprehensive Representation Learning for Resume-Job Matching"; Ma et al. 2022 "Unified Joint Matching Network").

**The solution in `aspects.py`:** the JD is segmented by EN+DE headers into must_have / nice_to_have / responsibilities / company / offer (→ `segment_jd`). Scoring is over section pairs, not whole documents:
- `cov_musthave_maxsim` = mean(max cosine(cand_skills_vec, each jd_phrase_vec)) — asymmetric coverage
- `cov_musthave_frac_above_tau` = the fraction of must-have phrases with maxsim > τ=0.6
- `n_musthave_missing` = the number of uncovered bullet points (keyword-based)
- `sim_experience_responsibilities`, `sim_skills_requirements`, `sim_domain_company`, `sim_fullcv_fulljd` — paired symmetric (backstop)
- `seniority_gap` (signed), `colbert_req_to_cv` (lazy MaxSim), `sparse_req_vs_cv` (lexical)
- Taste features from the PositiveLibrary (nearest/centroid/drift, company_overlap)
- Gate features: `lang_de_required`, `geo_distance_norm`, `is_remote_hint`, `recency_days`

**bge-m3:** `BGEM3FlagModel("BAAI/bge-m3", use_fp16=False, devices="cpu")` returns dense(1024) L2-normed + lexical_weights + colbert_vecs in one batch. `FlagEmbedding>=1.3` is the only new hard-dep (torch is already needed for bge). `FlagReranker("BAAI/bge-reranker-v2-m3", use_fp16=False)` for a cross-encoder re-rank of the top-K.

**PositiveLibrary** (transplanted from zotero-summarizer): P = vacancies with score_1_5≥4 OR applied=1 OR interview=1; an L2-normed (N,1024) matrix; leave-one-out against cosine=1 during training; cold start (P=∅) → taste features=0.

**`match_score`** (cold-start headline): `0.35*cov_musthave_maxsim + 0.15*cov_frac + 0.25*sim_exp_resp + 0.15*sim_skills_req + 0.10*sim_fullcv - 0.08*min(2,|gap|) if gap<0 - 0.025*min(6, n_missing)`, clip [0,1]. The full feature vector: `[dense_1024 ++ named_27] = 1051`.

### 7.2 Cheap ML triage gate (`triage.py`)

**Motivation:** LLM normalization costs ~150 calls/night. The aspect match_score gives an immediate pre-sort without the LLM; after ≥30 labels accumulate, a LightGBM regressor supersedes the fixed formula and learns the user's taste (Grinsztajn et al. 2022 "Why tree-based models still outperform deep learning on tabular data"; Joachims 2002 implicit feedback; Rendle 2009 BPR).

**Technique:**
- **Target** y ∈ [1,5]: score_1_5 from the label; applied=1 → 5.0; otherwise 2.0.
- **GroupKFold(n=5) by dedup_key**: cross-board duplicates don't leak from train into val.
- **Monotone constraints** on the coverage/similarity features (LightGBM `monotone_constraints`): makes a tiny model (n<50 rows) robust to overfitting (Chen & Guestrin 2016; Friedman 2001 GBM).
- **Isotonic calibration** (IsotonicRegression y_min=1, y_max=5, out_of_bounds=clip): applied only if it improves top-band macro-F1 — ranking is preserved by monotonicity.
- **Temporal holdout**: the newest 20% of groups by dedup_key → test (an honest estimate for a label time series; Blundell et al. 2015).
- **Metrics**: Spearman ρ, Kendall τ, NDCG@10, MAE, Cohen's κ over 4 buckets.
- **Cold start**: match_score, hard-drop off; rank-only. ≥30 labels → LGBM + drop the bottom bucket.
- **Bucket thresholds** (config): must≥4.5, should≥3.5, could≥2.0, drop<2.0. drop-Priority is excluded from the normalize queue → the LLM isn't spent on the worst.

### 7.3 Agentic search (`agent_runtime.py`, `sources/agent_discovery.py`, `investigate.py`)

**A ReAct agent** on kl_agent_builder + local qwen3:8b (ollama `/v1/`): WebSearchTool (DuckDuckGo keyless / SearXNG via `SEARXNG_URL`) + WebFetchTool. Two uses:

1. **Discovery crawler** (`scrape(cfg,con)`): a system prompt with the candidate profile, the task is to find postings beyond the aggregators and return a JSON array. Found vacancies → `source="agent"` → the same funnel (geo→hardfilters→dedup→features→triage→normalize→judge). Bounded `max_turns=8`, `max_results=20`.

2. **Top-match investigator** (`investigate_top(cfg,con,slate_date,top_n)`): for each of the top-N SCORED vacancies — the agent fetches the career page, verifies still-open + not-Zeitarbeit + culture signals, extracts verified requirements + salary. The result → an `investigation` sidecar table + a patch to `vacancy_feature`: `requirements_verified=1`, `company_known=1`, `salary_vs_target_gap`. On a re-run, `extract_features` recomputes coverage over the verified text.

**Robustness:** the KL_MODEL/KL_BASE_URL/KL_API_KEY env-overrides are cleared before creating the client. Both steps are non-fatal (`_step` in the pipeline). `parse_json_output` strips markdown fences before `json.loads`. Mock interface: `monkeypatch.setattr("schabasch.agent_runtime.build_agent", ...)` — the whole agentic layer is tested without kl_agent_builder.
