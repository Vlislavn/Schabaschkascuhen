# Обзор литературы и сравнение с SOTA

**Проект:** Schabaschkascuhen — персональный пайплайн поиска вакансий на немецком рынке
(JobSpy-форк + нативный клиент Arbeitsagentur → Normalizer (qwen3.5:4b) → жёсткие фильтры → LLM-Judge (qwen3:8b) → дневной слейт → метки Влада: ~100 bootstrap + ~10/день).
**Дата:** 2026-06-12. **Статус:** обязательный deliverable — lit-review + сравнение с SOTA.

---

## 1. Вопросы обзора

План проекта опирается на цепочку «скрейп → нормализация → фильтры → LLM-судья → слейт → обратная связь», при этом весь обучающий сигнал — это ~100 одноразовых меток и ~10 меток в день от одного человека. Из этого следуют конкретные вопросы, которые обзор должен был закрыть **до** фиксации роадмапа:

- **Q1. Надёжность LLM-судьи на локальных малых моделях.** Может ли qwen3:8b выдавать калиброванные оценки 1–5? Pointwise vs pairwise vs listwise; известные biases (позиция, многословность) и как их механически гасить; чем измерять калибровку.
- **Q2. Как потратить бюджет в ~100 меток.** Какая стратегия отбора примеров работает при низком бюджете (active learning «по-бедному»); какой формат метки надёжнее (шкала 1–5 vs best-worst); даёт ли что-то free-form элиситация предпочтений.
- **Q3. Проверенная архитектура job-рекомендаций.** Какая форма пайплайна доказанно работает (RecSys Challenge 2017 = канонический job-rec бенчмарк); какой embedding-бэкбон брать для смеси немецких и английских объявлений; нужна ли таксономия ESCO/ISCO.
- **Q4. Что обучать при n=100.** Файнтюнить ли что-либо LLM-размера; frozen embeddings + классическая модель vs SetFit vs нейроранкер.
- **Q5. AI-slop.** Реализуема ли локальная детекция машинно-сгенерированных текстов вакансий, и можно ли по ней фильтровать жёстко.
- **Q6. Import-over-build.** Какие репозитории форкать / импортировать как библиотеку / копировать как паттерн / отклонить — в соответствии с правилом «минимум писать самим».
- **Q7. Инструменты разметки.** Окупается ли готовая annotation-платформа (Argilla, Label Studio, doccano, Prodigy) при объёме 100 one-shot + 10/день, или достаточно xlsx + микро-FastAPI.
- **Q8. Сравнение с SOTA-паттернами.** Где наш план расходится с паттернами из локальной базы знаний (claude-code, ARE/Gaia2, aislop) по retry, error-таксономии, пиннингу судьи, slop-скорингу и дисциплине промптов.

### Решения, поддержанные литературой

1. **Протокол судьи — сравнительный, не pointwise.** На масштабе qwen 4b/8b listwise проваливается, а pointwise 1–5 некалиброван (PRP, Setwise): setwise-турниры группами 3–5 с pick best/worst; если 1–5 рубрика остаётся для отображения — калибровать якорными примерами из меток Влада и сверять каппой с pairwise-исходами.
2. **Механический дебиас каждого вызова судьи:** рандомизация и своп порядка с агрегацией (position bias до ~75% предпочтения первого слота), нормализация/усечение длины описаний (verbosity bias), постоянная метрика — каппа Коэна судья-vs-Влад на отложенных метках.
3. **Судья из двух стадий:** детерминированные/бинарные gate-проверки полей Normalizer (локация, remote, язык, тип контракта, сеньорность) **до** любого LLM-сравнения предпочтений — бинарные критерии не ложатся на pairwise-парадигму и тратят бюджет сравнений.
4. **Бюджет в 100 меток — low-budget-AL стилем:** кластеризация bge-m3 эмбеддингов и слейты из типичных/репрезентативных примеров со стратификацией по кластерам и бордам (TypiClust: uncertainty sampling ниже random при < нескольких сотен меток); uncertainty/margin — на потом, но low-confidence объекты судьи — в слейты (Personalized Judge).
5. **Метки как best-worst 4-туплы** (выбрать лучшего + худшего), не 1–5 звёзд — тот же бюджет, значимо выше надёжность, каждый тупл = 5 pairwise-ограничений (Kiritchenko & Mohammad); дополнить одноразовым GATE-интервью свободным текстом, которое становится стоящим контекстом судьи (GATE, Sanner).
6. **Обучаемый ранкер при n=100:** frozen bge-m3 фичи (cosine профиль→вакансия, mean/max similarity к понравившимся, ESCO-матч, метаданные) → LightGBM или регуляризованный логрег; без файнтюнинга LLM, без нейроранкера (Grinsztajn, SetFit, уроки MSR RecSys-2017); единственная подкреплённая эскалация — SetFit-style contrastive-тюнинг энкодера; негативы выбирать осознанно (показанная-но-пропущенная — слабый негатив).
7. **ESCO/ISCO как каркас:** Normalizer прикрепляет ESCO-оккупацию + топ-скиллы через embedding similarity; использовать как ключи стратификации слейтов, контекст судьи и помощь кросс-бордовому дедупу — но бэкбоном матчинга остаётся сырая bge-m3 близость (ESCOXLM-R, MELO).
8. **AI-slop — только soft-фича:** Binoculars-style perplexity-contrast скор двумя локальными моделями при фиксированном low-FPR пороге, только для описаний выше порога длины; никогда не hard-filter (RAID: катастрофические провалы при сдвиге домена; надёжная short-text детекция — closed-source по Jabarian & Imas); комбинировать с эвристиками кросс-постингового boilerplate/повторов.

### Где доказательная база тонкая (честные оговорки)

- **lmd_classi** — практикёрский бенчмарк одного автора на одном датасете (французские короткие тексты), не peer-reviewed; используем как подтверждающий, не основной аргумент.
- **Personalized Judge (~70% потолок согласия)** измерен на generic-персонах; наш режим (богатый first-person профиль + few-shot из меток того же человека) ближе к лучшему случаю статьи, но прямых измерений для job-domain нет — поэтому каппа судья-vs-Влад остаётся обязательной метрикой, а не предположением.
- **Binoculars на немецких вакансиях** — экстраполяция: и Binoculars, и RAID меряны на английском, job-ad boilerplate out-of-domain; порог придётся ревалидировать вручную на нашей выборке.
- **Jabarian & Imas** — working paper (NBER 2025), ещё без peer review.

### Пробелы нашего плана vs SOTA

1. **Нет пиннинга идентичности грейдера сверх rubric_version:** JudgeScore хранит имя модели, но не ollama digest и не хэш few-shot набора — дельты еженедельного CV agreement сейчас неатрибутируемы к (rubric | few-shot | model), нарушая are/pinned-judge-model.
2. **CV-гейт — одно pooled число:** нет fold-level STD/SEM, нет повторных прогонов (N=3) недетерминированного локального судьи, нет Pass^k-аналога — нарушает are/multi-run-variance; флаки-судья может пройти гейт 75%.
3. **Нет tri-state исходов:** «судья вернул invalid JSON → карточка остаётся normalized» смешивает unjudgeable с not-yet-judged в знаменателях воронки; нет harness-fault класса, отделяющего «Normalizer сломал карточку» от «судья ошибся» в еженедельной метрике agreement (are/failure-taxonomy).
4. **slop_flag бинарный в схеме card_json** — нет severity-weighted findings, нормализации по длине/плотности, smoothing, тиров (aislop/slop-scoring); длинные немецкие постинги будут смещены к slop по сырым счётчикам сигналов.
5. **Retry — ad-hoc проза** («retry x2 с backoff» в Extension 7a): нет центрального враппера, Retry-After не учитывается, нет jitter, нет таблицы retryability по статус-кодам, нет purpose-aware (batch-vs-interactive) бюджета (claude-code/retry-backoff).
6. **Логирование ошибок неструктурировано** («классифицированные ошибки в лог»): нет фиксированной coarse-enum колонки + lossless raw-details колонки в SQLite, поэтому FSM не может принимать типизированные retry-решения, а воронка не может считать классы ошибок (claude-code/error-envelope).
7. **Few-shot промпт судьи без дисциплины сборки:** нет структуры `<example>`+`<commentary>`, нет негативных примеров (score-1 / insufficient-evidence), нет программной интерполяции словаря тегов из констант схемы, нет flag-based смены наборов примеров по мере накопления меток (claude-code/tool-prompts-fewshot).
8. **profile/rubric.md без дисциплины staleness/supersession:** еженедельные обновления рискуют append-only ростом и противоречивыми фактами предпочтений; нет mtime-based «N days ago» свежести на few-shot метках (claude-code/memdir-file-memory).
9. **Canary-вердикты — бинарные min-row assertions** без закрытого статус-enum (dead_scraper / degraded / empty_market / stale_parser) — измеренная сигнатура Google silent-zero заслуживает собственного счётного типа отказа, а не alert-строки (are/failure-taxonomy).
10. **Нет детерминированного short-circuit до LLM-нормализации** (content-hash reuse для repost'ов, min-length floor) — ночной K-бюджет оплачивает повторную работу, которую предсказывает измеренный churn TBC-9 (дисциплина дешёвого выхода are/hard-before-soft-judge).

### Приоритет внедрения (топ-3 по вердикту SOTA-маппинга)

1. **Hard-before-soft фильтр-лестница** (are/hard-before-soft-judge, tool_judge.py#L591-L631) + content-hash short-circuits перед нормализацией — большинство вердиктов становится детерминированными/воспроизводимыми, оба LLM-бюджета капятся действительно-неоднозначным остатком.
2. **Пиннинг полного grader-tuple** (are/pinned-judge-model, configs.py#L27-L59) + сразу fold-level SEM и N=3 verdict-stability (are/multi-run-variance) — иначе CV-гейт и все тренды agreement суть неатрибутируемый шум.
3. **Типизированная tri-state таксономия отказов во всём FSM** (are/failure-taxonomy, judgment.py#L13-L104 + claude-code/error-envelope) — воронка становится счётными метриками, agreement судьи остаётся честным. Почётное упоминание: density-smoothed slop-скоринг с тирами вместо бинарного slop_flag (aislop/slop-scoring) — дёшев при проектировании схемы карточки, дорог при ретрофите после сбора bootstrap-меток.

## 6. Сводка: что обзор меняет в плане

- **Судья: отказаться от listwise-слейтов и сырого pointwise 1–5 на qwen3:8b** — перейти на setwise/pairwise-сравнения (группы 3–5, pick best/worst) с рандомизацией и свопом порядка кандидатов, нормализацией длины описаний и калибровкой 1–5-рубрики якорными примерами; стоящая метрика калибровки — каппа Коэна судья-vs-Влад на отложенных метках (PRP, Setwise, Gu et al.).
- **Разделить судью на два этапа:** детерминированные бинарные gate-проверки жёстких ограничений (локация, remote, язык, Zeitarbeit, тип контракта) ДО любого LLM-сравнения; маршрутизация полей через декларативный CheckerType-enum (hard vs soft), а не if/else (Eugene Yan + are/hard-before-soft-judge).
- **Жёсткие детекторы строить с FP-гардами:** regex-детектор немецкого требования = broad pattern + exoneration-каскад («nice to have», «is a plus», «von Vorteil» → not-required), scope-гейт boilerplate-секций до матчинга, регрессионный гейт ≥25/27 на 27-строчном spike-тестсете; дедуп — exoneration-first guard-stack, fuzzy-кандидаты логируются и никогда не auto-мержатся (aislop/fp-guards).
- **Сбор меток: best-worst 4-туплы вместо 1–5 звёзд** (каждый тупл = 5 pairwise-ограничений); первые слейты — TypiClust-стилем: кластеризация bge-m3 эмбеддингов, типичные примеры, стратификация по бордам/ролям/языку; uncertainty sampling отложить до нескольких сотен меток; low-confidence вердикты судьи направлять в следующий разметочный слейт (Kiritchenko & Mohammad, TypiClust, Personalized Judge).
- **Добавить одноразовое GATE-интервью** (LLM задаёт Владу открытые вопросы и edge cases) и хранить транскрипт + письменный профиль предпочтений как стоящий контекст судьи; система должна приемлемо ранжировать zero-shot до любых меток (GATE, Sanner).
- **Learned-ранкер при n=100: frozen bge-m3 + LightGBM/регуляризованный логрег** (cosine профиль→вакансия, mean/max similarity к понравившимся, ESCO-матч, метаданные); никакого файнтюнинга LLM/cross-encoder; единственная подкреплённая эскалация — SetFit-style contrastive-тюнинг энкодера; негативы выбирать осознанно — показанная-но-пропущенная вакансия — слабый негатив (Grinsztajn, SetFit, MSR RecSys-2017).
- **ESCO/ISCO как каркас, не зависимость:** Normalizer присваивает ESCO-оккупацию + топ-скиллы через embedding similarity; использовать как ключи стратификации слейтов, контекст судьи и помощь дедупу — бэкбон матчинга остаётся raw bge-m3 (ESCOXLM-R, BGE M3).
- **slop_flag: заменить бинарный флаг на density-smoothed 0–100 score с тирами clean/suspicious/slop** (severity-веса, нормализация по длине + smoothing, log1p); Binoculars-style perplexity-contrast только как soft-фича выше порога длины описания; никогда не hard-filter; внедрить в схему карточки ДО сбора bootstrap-меток — ретрофит после дорог (aislop/slop-scoring, Binoculars, RAID, Jabarian & Imas).
- **Import-лист v1:** (1) cherry-pick JobSpy PR #347 — 13 строк, оживляющих Glassdoor; (2) перевести AA-клиент на /pc/v4/jobdetails/{base64(refnr)} по bundesAPI/jobsuche-api — закрывает 0% описаний; (3) RapidFuzz + паттерн JobFunnel filters.py для дедупа; (4) llm-rankers в setwise/pairwise-режиме для ранжирования слейта; (5) скопировать UI-цикл QuickApply + скелет VincenzoImp/job-search-tool для digest-страницы. Дополнительно: test-merge PR #366 (Xing), кандидаты PR #354 и #306, ослабить пин markdownify (issue #349). Google Jobs — неподдерживаем в v1 (фикса upstream нет).
- **Разметка: план не меняется** — Surface 1 = генерируемый xlsx через openpyxl (DataValidation-дропдауны, frozen panes, hyperlink на job_url, re-ingest через pandas.read_excel в SQLite), Surface 2 = ~100-строчная FastAPI-страница, скопированная из VerdictPanel zotero-summarizer; annotation-платформы не окупаются при нашем объёме; ревизия (Label Studio CE → Prodigy) только при >1 аннотатора или >~100 карточек/день.
- **Пиновать полный grader-tuple:** DEFAULT_JUDGE_MODEL + ollama digest + rubric_version + fewshot_set_hash в каждой строке JudgeScore; ручки моделей Normalizer и Judge разделить, чтобы бамп qwen3.5:4b не менял грейдер тихо (are/pinned-judge-model).
- **CV-гейт ≥75% пересчитать честно:** mean ± SEM по fold-level средним (ddof=1, guard n≤1), N=3 прогона судьи на карточку с Pass^k-style verdict-stability; unjudgeable карточки вне знаменателя agreement; флаки-судья не должен проходить launch-гейт (are/multi-run-variance).
- **Типизированная tri-state таксономия отказов во всём FSM:** закрытый enum причин на месте отклонения (filtered_remote / filtered_geo / filtered_language_de / filtered_temp_agency / llm_invalid_json / details_403 / expired_404...), отдельный harness-fault класс (broken-card Normalizer не дефлирует agreement судьи); canary-вердикты — closed enum: dead_scraper (сигнатура Google silent-zero) / degraded / empty_market / stale_parser (are/failure-taxonomy).
- **Классифицированный error envelope в SQLite:** колонка last_error_class (закрытый enum {invalid_json, schema_violation, empty_output, timeout, endpoint_down, oversized_input, unknown}) + lossless last_error_details; FSM-решения о retry читают enum, никогда не regex по логу; JSON-парсинг вывода qwen лояльный, но fail-closed (claude-code/error-envelope).
- **Один центральный with_retry-враппер** для Desc-Fetcher и всех ollama-вызовов: Retry-After приоритетно, capped exponential backoff + 0–25% jitter, таблица retryability по статусам (AA 403/empty-body = non-retryable + canary-инкремент), purpose-aware бюджет для ночного батча 03:00; плюс детерминированные short-circuits до LLM (content-hash reuse карточек для repost'ов, min-length floor → undescribable) для экономии ночного K-бюджета (claude-code/retry-backoff, are/hard-before-soft-judge).
- **Промпт судьи собирать в коде:** блоки `<example>`+`<commentary>` из свежих экстремальных меток Влада, негативные примеры (score 1 + repeller; insufficient-evidence → low-confidence), интерполяция словаря тегов из тех же констант, что и SQLite-схема, один литеральный JSON-пример после правила схемы; профиль предпочтений вести memdir-style — файл-на-факт с frontmatter, supersession in place вместо append, возраст меток «N days ago» в few-shot заголовке (claude-code/tool-prompts-fewshot, memdir-file-memory).

## 7. v2: аспектное CV↔JD-матчинг + ML-гейт + агентный поиск (реализовано 2026-06-14)

### 7.1 Аспектное матчинг (bge-m3, `aspects.py` + `features.py`)

**Проблема одного blob-косинуса:** whole-document cosine резюме↔вакансия — это мера *похожести*, а не *пригодности*: «About us» с похожим доменом накачивает score независимо от реальных требований. SOTA-подход для job-skill matching — **асимметричное покрытие must-have** (покрывают ли скиллы кандидата конкретные требования JD?), а не симметричный blob (Xu et al. 2023 «Towards Comprehensive Representation Learning for Resume-Job Matching»; Ma et al. 2022 «Unified Joint Matching Network»).

**Решение в `aspects.py`:** JD сегментируется по EN+DE заголовкам на must_have / nice_to_have / responsibilities / company / offer (→ `segment_jd`). Скоринг — пары секций, а не целые документы:
- `cov_musthave_maxsim` = mean(max cosine(cand_skills_vec, each jd_phrase_vec)) — асимметричное покрытие
- `cov_musthave_frac_above_tau` = доля must-have фраз с maxsim > τ=0.6
- `n_musthave_missing` = число непокрытых bullet-пойнтов (keyword-based)
- `sim_experience_responsibilities`, `sim_skills_requirements`, `sim_domain_company`, `sim_fullcv_fulljd` — парные симметричные (backstop)
- `seniority_gap` (signed), `colbert_req_to_cv` (lazy MaxSim), `sparse_req_vs_cv` (lexical)
- Taste-фичи из PositiveLibrary (nearest/centroid/drift, company_overlap)
- Gate-фичи: `lang_de_required`, `geo_distance_norm`, `is_remote_hint`, `recency_days`

**bge-m3:** `BGEM3FlagModel("BAAI/bge-m3", use_fp16=False, devices="cpu")` возвращает dense(1024) L2-normed + lexical_weights + colbert_vecs за один батч. `FlagEmbedding>=1.3` — единственная новая hard-dep (torch уже нужен bge). `FlagReranker("BAAI/bge-reranker-v2-m3", use_fp16=False)` для cross-encoder re-rank top-K.

**PositiveLibrary** (transplant из zotero-summarizer): P = вакансии с score_1_5≥4 OR applied=1 OR interview=1; матрица L2-normed (N,1024); leave-one-out против косинуса=1 при обучении; cold start (P=∅) → taste-фичи=0.

**`match_score`** (cold-start headline): `0.35*cov_musthave_maxsim + 0.15*cov_frac + 0.25*sim_exp_resp + 0.15*sim_skills_req + 0.10*sim_fullcv - 0.08*min(2,|gap|) if gap<0 - 0.025*min(6, n_missing)`, clip [0,1]. Полный вектор признаков: `[dense_1024 ++ named_27] = 1051`.

### 7.2 Cheap ML triage gate (`triage.py`)

**Мотивация:** LLM-нормализация стоит ~150 вызовов/ночь. Аспектный match_score даёт сразу предсортировку без LLM; после накопления ≥30 меток LightGBM-регрессор вытесняет фиксированную формулу и учится вкусу пользователя (Grinsztajn et al. 2022 «Why tree-based models still outperform deep learning on tabular data»; Joachims 2002 implicit feedback; Rendle 2009 BPR).

**Техника:**
- **Цель** y ∈ [1,5]: score_1_5 из label; applied=1 → 5.0; иначе 2.0.
- **GroupKFold(n=5) по dedup_key**: кросс-борд дубли не перетекают из train в val.
- **Monotone constraints** на coverage/similarity фичах (LightGBM `monotone_constraints`): делает крошечную модель (n<50 строк) устойчивой к переобучению (Chen & Guestrin 2016; Friedman 2001 GBM).
- **Isotonic calibration** (IsotonicRegression y_min=1, y_max=5, out_of_bounds=clip): применяется, только если улучшает top-band macro-F1 — ranking сохраняется монотонностью.
- **Temporal holdout**: newest 20% групп по dedup_key → тест (честная оценка для временных рядов меток; Blundell et al. 2015).
- **Метрики**: Spearman ρ, Kendall τ, NDCG@10, MAE, Cohen's κ по 4 бакетам.
- **Cold start**: match_score, hard-drop выключен; rank-only. ≥30 меток → LGBM + drop нижнего бакета.
- **Bucket thresholds** (config): must≥4.5, should≥3.5, could≥2.0, drop<2.0. drop-Priority исключается из normalize-очереди → LLM не тратится на худших.

### 7.3 Аgentic search (`agent_runtime.py`, `sources/agent_discovery.py`, `investigate.py`)

**ReAct-агент** на kl_agent_builder + local qwen3:8b (ollama `/v1/`): WebSearchTool (DuckDuckGo keyless / SearXNG via `SEARXNG_URL`) + WebFetchTool. Два применения:

1. **Discovery crawler** (`scrape(cfg,con)`): system prompt с профилем кандидата, задача — найти постинги за пределами агрегаторов, вернуть JSON-массив. Найденные вакансии → `source="agent"` → та же воронка (geo→hardfilters→dedup→features→triage→normalize→judge). Bounded `max_turns=8`, `max_results=20`.

2. **Top-match investigator** (`investigate_top(cfg,con,slate_date,top_n)`): для каждого из top-N SCORED вакансий — agent fetches career page, verifies still-open + not-Zeitarbeit + culture signals, extracts verified requirements + salary. Результат → `investigation` sidecar table + patch `vacancy_feature`: `requirements_verified=1`, `company_known=1`, `salary_vs_target_gap`. При повторном запуске `extract_features` пересчитывает coverage по верифицированному тексту.

**Robustness:** env-override KL_MODEL/KL_BASE_URL/KL_API_KEY очищаются перед созданием клиента. Оба шага non-fatal (`_step` в pipeline). `parse_json_output` стрипает markdown-фенсы перед `json.loads`. Мок-интерфейс: `monkeypatch.setattr("schabasch.agent_runtime.build_agent", ...)` — весь агентный слой тестируется без kl_agent_builder.
